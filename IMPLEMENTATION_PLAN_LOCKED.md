| CORS | Exact comma-separated allow-list; no wildcard credentials | Meets public/private boundary requirements. |

## 4. Conflict and gap resolutions

1. The OpenAPI server prefix makes `/api/v1/health` canonical. A root `/health` alias exists only for Render health checks and returns the same safe payload; it is recorded in `DECISIONS_IMPLEMENTED.md` as provider plumbing, not a second API contract.
2. Public and admin reads share several OpenAPI paths. An optional valid bearer token selects admin DTOs; anonymous responses use explicit allow-lists. Optional private fields are omitted publicly. Private sessions return 404.
3. `WorkoutSession.notes_private` exists in the contract for authenticated responses but is never populated in anonymous responses.
4. The data schema permits readiness and expanded settings, while the initial OpenAPI omits write routes for readiness and some settings. Database support is implemented, but no undocumented public route is invented. Any necessary owner UI extension is added contract-first in LVL-104 and recorded.
5. Split duplication and reorder use the existing `GET` + `POST`/full-tree `PATCH` contract; activation and archival use their dedicated endpoints.
6. The seed file's optional-day `modes` are normalized into one optional day whose initial items are the conditioning mode; specialization variants remain discoverable through alternatives/library rather than creating undocumented day subtypes.
7. The active upper/lower schedule maps Monday/Tuesday/Thursday/Friday to sequences 1–4 and Saturday to optional sequence 5. Other days are rest days. This deterministic rule is configurable in seed code and documented.
8. Public hidden water returns `null` inside the dashboard. Direct anonymous hydration access returns 404 so visibility cannot be enumerated.
9. Session volume is the sum of working-set `load_kg × reps`; bodyweight, duration, distance, and rounds use their dedicated record types and do not fabricate load volume.
10. Estimated 1RM uses Epley only for working load/reps sets with 1–12 reps, is labelled as estimated, and never drives a maximal-single suggestion.
11. The handoff requests Tailwind or CSS Modules; centralized token CSS with feature-scoped class naming satisfies the centralized styling requirement without a framework dependency.
12. The handoff files remain in their supplied nested directory. Runtime copies of OpenAPI and seeds are synchronized from that source rather than relocating or rewriting the owner handoff.

All later deviations require an entry in `DECISIONS_IMPLEMENTED.md`, a contract/schema test where applicable, and a ticket-specific PR note.

## 8. Pull-request and commit gate

For every row above:

1. `git switch main && git pull --ff-only origin main`
2. create the exact listed branch;
3. implement only its allowed scope;
4. run ticket-relevant lint, types, unit/integration/contract/browser checks;
5. commit with `<type>(<scope>): <imperative summary>` and `Refs:`/`AC:` footers;
6. rebase on `origin/main`, push, and open a complete PR using `.github/pull_request_template.md`;
7. require relevant CI, resolve review, and squash merge with a Conventional Commit title;
8. delete local and remote branch;
9. record commit hash and PR URL in `DEPLOYMENT_STATUS.md` or the final verification report.

No dependent ticket starts from an unmerged branch. Failed checks block merge. `--force-with-lease` is allowed only after rebasing the currently owned ticket branch; `main` is never force-pushed.

## 13. Definition of done

LEVELS is done only when every ticket PR is merged, all required commands pass from a clean checkout, the production database and both deployments are healthy, the complete public/admin browser journey succeeds, no private field or secret is exposed, and the final reports contain evidence plus exact main commit, release tag, deployment URLs, and PR links. A provider authorization blocker is reported as incomplete rather than described as success.
